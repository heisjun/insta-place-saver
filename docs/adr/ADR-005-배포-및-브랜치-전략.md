# ADR-005: 배포 및 브랜치 전략

## 배경

개인 프로젝트이나 체계적인 이력 관리와 안정적인 배포를 위해 브랜치 전략이 필요하다.

## 결정

**GitHub Flow** 방식 채택

```
feature/xxx → PR → main (Vercel 자동 배포)
```

- `main`: 항상 배포 가능한 상태 유지. Vercel Production 브랜치.
- `feature/step-N-xxx`: 기능 단위 작업 브랜치
- PR 없이 main 직접 push 금지

## 커밋 컨벤션

**Conventional Commits** 사용

| 타입 | 설명 |
|---|---|
| `feat` | 새 기능 |
| `fix` | 버그 수정 |
| `chore` | 설정, 패키지, 빌드 관련 |
| `refactor` | 기능 변경 없는 코드 개선 |
| `docs` | 문서 작성/수정 |

**PR 제목 형식:** `[Step N] type(scope): 설명`

예시:
```
[Step 5] feat(map): 카카오맵 지도 페이지 구현
[Step 6] feat(places): 장소 목록 페이지 구현
```

## Vercel 설정

- **Production Branch**: `main`
- PR 머지 → Vercel 자동 프로덕션 배포
- PR 오픈 상태 → Vercel Preview 배포 (별도 URL 생성)

## 환경변수 관리

- `.env.local`: 실제 키 (gitignore, 절대 커밋 금지)
- `.env.local.example`: 키 목록만 (xxx로 마스킹, 커밋됨)
- Vercel 대시보드: 프로덕션 환경변수 별도 관리
