# dev-cycle — 개발 한 사이클 오케스트레이터

## 목적

한 기능 단위(컴포넌트·훅 하나, 혹은 작은 feature)의 작업을 다음 순서로 자동 진행한다.

```
[1] scaffold-component / scaffold-hook   ─→  골격·테스트 스캐폴딩
                ↓
[2] (사용자가 구현)
                ↓
[3] self-review                          ─→  4축 자체 검토 리포트
                ↓
[4] (blocker 0개면)
                ↓
[5] commit-and-pr                        ─→  커밋 메시지·PR 본문 초안
                ↓
[6] (사용자 승인 후 실행)
```

## 판단 기준 — 각 단계 진입 조건

| 단계 | 진입 판단 |
|---|---|
| `scaffold-*` | 작업명에 새 컴포넌트/훅 이름이 명시되어 있거나, 기존 코드와 충돌하지 않을 때만 실행. 이미 같은 이름의 파일이 있으면 중단하고 사용자에게 확인. |
| `self-review` | 사용자가 "구현 끝났어", "리뷰해줘", "검토해줘" 등으로 신호를 주거나, 마지막 사용자 메시지 이후 코드 변경이 누적되었을 때 호출. |
| `commit-and-pr` | `self-review` 결과 **blocker가 0개**일 때만 자동 진행. 1개 이상이면 사용자에게 "이슈 N개가 있는데 그대로 진행할까요?" 로 확인 받고 진행. |

## scaffold 스킵 판단 — *언제 1단계를 건너뛰는가*

`scaffold-component`와 `scaffold-hook`은 **React 컴포넌트·훅 전용**이다. 다음 작업 유형은 스캐폴딩 대상이 아니므로 **명시적으로 스킵**한다.

| 작업 유형 | 스킵 이유 | 그래도 해야 할 일 |
|---|---|---|
| 서버 유틸 (`lib/*.ts` 중 `"use client"` 없는 모듈) | React 훅이 아니라 scaffold-hook 부적합 | `*.test.ts` 단위 테스트는 *수동으로* 함께 작성 |
| API 라우트 (`src/app/api/*/route.ts`) | Next.js 규약 파일이라 스캐폴딩 형식이 다름 | 라우트 핸들러는 통합 테스트로 별도 검증 |
| SQL 마이그레이션 (`supabase/*.sql`) | 코드가 아니라 스키마 변경 | PR 본문에 *수동 실행 단계*를 배포 전 필수 작업에 명시 |
| 기존 파일 수정만 (버그 수정, 리팩토링) | 새 파일이 없으니 스캐폴딩 불필요 | 변경 범위에 비례해 테스트 추가·갱신 |
| 설정 파일 (`next.config.ts`, `eslint.config.mjs` 등) | 1회성 설정 | 필요 시 변경 동작을 별도 PR에서 검증 |

**스킵 시 보고 의무**: 1단계 진입 시점에 "이 작업은 X에 해당하므로 scaffold-* 단계를 건너뜁니다. 대신 Y는 직접 작성합니다" 라고 *명시적으로* 사용자에게 보고. 그래야 사용자가 "테스트를 안 만든 게 아니라 의식적으로 스킵했다"는 사실을 알 수 있다.

**잘못된 스킵**: 새 React 컴포넌트/훅을 만들면서 "간단해서" 스캐폴딩 생략 — 금지. 이 경우는 *반드시* 스캐폴딩 호출. 단순할수록 테스트가 5분이면 끝남.

## UI 컴포넌트 작업 시 보강 흐름

React 컴포넌트가 새로 들어가는 사이클이면, `scaffold-component` 호출 *전*에 다음 두 단계를 거친다.

### 1. shadcn 우선 검토
`.mcp.json`에 shadcn MCP가 등록되어 있으면 자연어로 "이 UI에 쓸 만한 shadcn 컴포넌트가 있는지" 먼저 묻는다. 있으면 *직접 만들지 않고* `npx shadcn@latest add <component>`로 가져와 사용. 이유:
- 표준 컴포넌트는 a11y·키보드 핸들링이 이미 검증돼있음
- 디자인 일관성이 자연스럽게 따라옴
- 우리가 만든 self-review 4축 중 *접근성*·*불필요 리렌더링* 두 축의 위험이 크게 줄음

shadcn 컴포넌트로 충분하면 scaffold-component 단계는 *생략*하고 self-review로 직행.

### 2. 직접 작성할 때만 frontend-design 적용
shadcn에 없거나 부족해서 *직접* 컴포넌트를 작성해야 할 때만 `frontend-design` 스킬의 가이드(대담한 톤 선택, generic 폰트·색 금지, 의도 있는 디자인)를 적용. 단순 wrap·composition은 생략 가능.

### 책임 분리 요약
| 도구 | 책임 |
|---|---|
| shadcn MCP | "이미 만들어진 표준 컴포넌트가 있는가" 판단·설치 |
| `frontend-design` | 직접 만들 때의 *미적 방향* |
| `scaffold-component` | 위치·Server/Client·테스트 골격 |
| `self-review` | 4축 검토 (a11y·리렌더링 등 모든 결과물에 적용) |

## 입력

- 자유 형식 자연어 (예: "PlaceFilterButton 컴포넌트 만들고 한 사이클 진행해줘")
- 또는 단계 명시 (예: "self-review만 돌려줘")

## 출력

각 단계 종료마다:
- 단계 이름 + 결과 요약 (3줄 이내)
- 다음 단계로 자동 진행할지, 사용자 승인을 받을지 명시

## 체이닝 규칙

- **자동 진행**: `scaffold-*` → 사용자 구현 대기 (자동 진행 X)
- **자동 진행**: `self-review` → blocker 0이면 `commit-and-pr` 호출
- **수동 진행**: `commit-and-pr` 결과는 *항상* 초안만 제시. 실제 `git commit` / `gh pr create`는 사용자 승인 후에만 실행.

## 사용 예시

```
사용자: "MemoEditor 컴포넌트 추가하고 한 사이클"
  → scaffold-component 호출 (MemoEditor)
  → 골격·테스트 생성 후 대기
사용자: (구현 완료) "검토하고 PR 올려줘"
  → self-review 호출
  → blocker 0 확인 후 commit-and-pr 호출
  → 초안 제시, 승인 대기
사용자: "응 진행해"
  → git commit + gh pr create 실행
```
